# alembic implementer

Write migrations that are reversible and safe against live data. Delegate API reality to the
official Alembic docs; own the safety patterns below. Migrations are `risk:high` by default.

## Online-safe patterns (Postgres)

**Add NOT NULL to a populated table** — three revisions, never one:
1. add the column nullable; 2. backfill (batched if >100k rows); 3. `ALTER COLUMN … SET NOT
NULL`.

**Add an index on a large table** — `CREATE INDEX CONCURRENTLY`, which must run **outside a
transaction** (`with op.get_context().autocommit_block():`).

**Add an enum value** — `ALTER TYPE … ADD VALUE` is non-transactional in PG < 12; isolate it in
its own revision. Postgres can't drop enum values — document that in downgrade.

**Rename a column on a live app** — expand/contract (add new, backfill, switch reads/writes,
drop old) across revisions; a bare rename breaks in-flight requests.

## Decision framework

- **One migration or several?** Any change that locks a large table or is non-transactional
  gets its **own** revision. Never bundle a concurrent index or enum add with other DDL.
- **Autogenerate or hand-write?** Autogenerate is a **draft** — it misses server defaults, enum
  changes, and index concurrency. Always read and correct it; never commit it unread.

## Review checklist (a reviewer checks against this — foolery review or human)

- [ ] Every `upgrade` has a real `downgrade`; irreversibility is documented, not faked.
- [ ] Autogenerated output was reviewed and corrected (defaults, enums, concurrency).
- [ ] NOT NULL on a populated table is split into nullable → backfill → set-not-null.
- [ ] Large-table index uses `CONCURRENTLY` outside a transaction.
- [ ] No destructive DDL (drop column/table) without an explicit, confirmed bead.
- [ ] One logical change per revision; message is clear.
- [ ] Multi-db/federation runs with the correct `alembic -c <ini>`.

## Execution discipline

Test both directions on a scratch DB before the bead is done: `alembic upgrade head` then
`alembic downgrade -1`. Implement only the bead's `## Acceptance`; respect `## Out of scope`.
Read and obey `.product/principles.md`. File `discovered-from` beads for surprises.

## Handoffs

Model/schema shape → `pydantic` / `fastapi`. Output: the revision(s) + confirmation both
directions ran clean.
